Example 1

Query Builders in Modifiers

Runtime will no longer resolve Builder instances when used inside modifiers. Before, the Runtime would attempt to collapse all instances by calling ->get().

Result:

Example 2

Nested Query Builders

The previous checks to allow Builder instances to be used as variables (`builder_field.0.title) were overly aggressive. This stopped builders from being resolved if they were nested inside other arrays (note: the fix here is that it now properly calls the Query tag's index method. This nested array is built inside AppServiceProvider).

Result:

{{ nested:the_topics }} {{ title }}{{ unless last }},{{ /unless }} {{ /nested:the_topics }}

{{ noparse }}
1{{ nested:the_topics }}2 {{ title }}{{ unless last }}, {{ /unless }}3{{ /nested:the_topics }}
{{ /noparse }}
Example 3

Nested Query Builders with Parameters

This is similar to Example 2, but this was pretty much completely broken before. When switching to the previous behavior, you should receive a modifier not found exception.

Result:

{{ nested:the_topics order_by="title:desc" }} {{ title }}{{ unless last }},{{ /unless }} {{ /nested:the_topics }}

Example 4

Scoped Query Builders

The previous data existence check would not check the Cascade if something returned null. This would impact query builders since it is the existence check that sets things up to be sent through the Query tag. Before this example would simply do nothing.

Result:

{{ scope:block }} {{ the_topics order_by="title:desc" }} {{ title }}{{ unless last }},{{ /unless }} {{ /the_topics }} {{ /scope:block }}

Next
{{ noparse }}
1{{ scope:block }}2 {{ the_topics order_by="title:desc" }}3 {{ title }}{{ unless last }}, {{ /unless }}4 {{ /the_topics }}5{{ /scope:block }}
{{ /noparse }}
Example 5

Query Builders as Tag Parameters

Like with modifiers, the Runtime will now leave Builder instances alone when being supplied to tags using dynamic binding.

Result:

{{ if {is_builder :tags="the_topics"} }} The tag received a query builder! {{ /if }}
